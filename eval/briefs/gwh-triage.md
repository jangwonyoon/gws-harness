# Eval Brief: `/gwh:triage`

## 스킬 목적 (주장)
Gmail 미읽음 메일을 **중요도별 4단계(🔴 긴급 / 🟡 액션 필요 / 🟢 FYI / ⚪ 무시)** 로 분류하고, 액션 아이템을 추출하여 `~/.gwh/triage-{date}.md`에 저장한다.

## 해결하겠다는 문제
- 쏟아지는 메일 중 **실제로 액션이 필요한 것만 빠르게 찾기**
- 수동 트리아지 시 반복되는 "열어보고 → 중요도 판단 → 회신 결정" 사이클 자동화

## 평가 시나리오 (모두 실행)

### S1. 일상 트리아지 (Happy Path)
1. 미읽음 메일이 20-100통 쌓인 상태로 `/gwh:triage` 실행
2. 출력 확인:
   - 🔴/🟡 분류가 정확한가? (직접 메일을 열어보고 비교)
   - 액션 아이템이 실제 메일 내용과 일치하는가?
   - ⚪로 분류된 메일이 정말 무시해도 되는가?

### S2. 대량 미읽음 (Edge Case)
1. 미읽음 200통+ 상태에서 실행
2. `--max 50` 제한이 작동하는지 확인
3. 출력에 "약 N+통 중 최근 50통 분석" 같은 추정 표기 있는지 확인

### S3. 재실행 (Cache Behavior)
1. 같은 날 `/gwh:triage`를 두 번 실행
2. 두 번째 실행 시 "오늘 이미 트리아지 결과가 있습니다" 같은 캐시 확인 질문이 나오는가?
3. 캐시 재사용 시 Gmail API 호출이 일어나지 않는지 (시간 측정: `time` 명령)

### S4. 산출물 검증 (Composability)
1. `~/.gwh/triage-{today}.md` 파일 열기
2. 구조 확인: 🔴/🟡/🟢/⚪ 섹션 + 액션 요약
3. 다른 스킬(`/gwh:brief`, `/gwh:mail-to-ticket`)이 이 파일을 읽어 재사용 가능한지 확인

## 측정해야 할 지표

| 지표 | 측정 방법 |
|------|----------|
| 분류 정확도 | 🔴/🟡 분류된 메일 중 본인이 동의하는 비율 |
| 거짓 양성 | 🔴/🟡로 분류됐는데 실제로는 무시해도 되는 비율 |
| 거짓 음성 | ⚪로 분류됐는데 실제로는 액션 필요한 비율 |
| 실행 시간 | `time /gwh:triage` (Cache miss 기준) |
| 캐시 속도 향상 | Cache miss 시간 / Cache hit 시간 |

## 루브릭 적용 가이드

- **D1 문제 해결력**: 실제 쏟아지는 메일에서 액션 필요한 것만 걸러내는가?
- **D2 신호/노이즈**: 분류 결과의 정확도 + 장식 문구 비율
- **D3 컨텍스트 절약**: 수동으로 100통 트리아지 시 ~30분 vs 스킬 사용 시?
- **D4 재사용성**: triage 캐시가 brief/mail-to-ticket에서 재사용되는가?
- **D5 채택 확률**: 매일 아침 이걸로 메일 정리를 실제로 할 것인가?

## 관찰 포인트

- 회사 계정(업무 메일)과 개인 계정(뉴스레터 위주)에서 결과 품질이 다를 수 있음
- CodeRabbit/GitHub 알림처럼 자동 발송이지만 액션 필요한 메일의 분류 정확도 주목
- 한국어 메일 본문이 많을 때 영어 위주 분류 기준(urgent, review)이 잘 맞는지
