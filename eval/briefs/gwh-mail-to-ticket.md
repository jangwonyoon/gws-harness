# Eval Brief: `/gwh:mail-to-ticket`

## 스킬 목적 (주장)
특정 메일에서 **요구사항/버그를 추출**하여 Jira 이슈 또는 Notion 페이지를 생성한다. 메일 유형(버그/기능 요청/작업 요청)을 자동 감지하고 구조화.

## 해결하겠다는 문제
- 메일로 온 요청을 티켓화할 때 복붙 + 포맷팅 작업 반복
- 재현 단계/심각도 같은 필드를 수동으로 채우는 비효율

## 평가 시나리오

### S1. 버그 리포트 메일 (Happy Path)
1. triage 결과에서 버그 관련 메일 선택 (또는 메일 ID 직접 지정)
2. `/gwh:mail-to-ticket` 실행
3. 구조화 결과 확인:
   - 유형: 버그 리포트로 올바르게 감지?
   - 재현 단계가 메일 본문에서 정확히 추출?
   - 심각도 추정이 합리적?

### S2. 기능 요청 메일
1. 클라이언트/팀원의 기능 추가 요청 메일 선택
2. Epic + Story 분해 제안이 나오는가
3. SP 추정이 포함되는가

### S3. 대안 판단 검증 (⚠️ 핵심)
1. 단순한 린트 이슈/사소한 작업성 메일 선택
2. 스킬이 "이 경우는 티켓보다 PR 직접 수정이 낫다"는 **대안 판단**을 제공하는가
3. 맹목적으로 티켓 만들지 않는지 (오버엔지니어링 방지)

### S4. MCP 미연결 폴백
1. Jira MCP, Notion MCP 모두 미연결 상태에서 실행
2. 마크다운 형식으로 출력되는가 (사용자가 수동 복사용)
3. 에러 없이 graceful degradation

### S5. 승인 게이트
1. 구조화 결과 확인 후 **거절** 선택
2. Jira/Notion에 실제로 생성되지 않는지 직접 확인
3. 산출물 파일(`~/.gwh/ticket-*`)만 저장되는지

### S6. 실제 생성 (선택적, 테스트 프로젝트에서)
1. 테스트 Jira 프로젝트에서 실제 생성
2. ADF 포맷팅이 Jira에서 정상 렌더링되는가
3. 원본 메일 링크가 코멘트/description에 첨부되는가

## 측정해야 할 지표

| 지표 | 측정 방법 |
|------|----------|
| 유형 감지 정확도 | 자동 감지한 유형 vs 실제 의도 |
| 필드 추출 완성도 | 제목/설명/재현/심각도 중 몇 개가 유용하게 채워졌나 |
| 오버엔지니어링 회피 | S3에서 대안 판단이 나오는 비율 |
| 승인 전 생성 사고 | S5에서 거절했는데 만들어지면 **치명적** |

## 루브릭 적용 가이드

- **D1 문제 해결력**: 메일 → 티켓 변환 시간이 실제로 단축되는가?
- **D2 신호/노이즈**: 구조화된 결과가 바로 승인 가능한 품질인가
- **D3 컨텍스트 절약**: 수동 티켓 작성 10분 vs 스킬 사용 시
- **D4 재사용성**: `~/.gwh/ticket-*` 파일이 나중에 참고되는가
- **D5 채택 확률**: 메일로 요청이 올 때마다 이걸로 처리할 것인가?

## 관찰 포인트

- 한국어 메일 본문의 구조화 품질 (영문 키워드 감지가 한글에서도 작동?)
- 심각도 자동 판단이 과도하게 보수적이지 않은가
- ADF 포맷팅 품질 (Jira에서 목록/코드블록이 제대로 렌더되는지)
- 대안 판단이 일관되게 제공되는가 (맹목적 생성 방지)

## ⚠️ 쓰기 스킬 주의사항

이 스킬은 Jira/Notion에 **실제 티켓/페이지를 생성**한다:
- 테스트 프로젝트/데이터베이스에서만 실제 생성
- 프로덕션 Jira에서는 S5 거절까지만 평가
- 실수로 생성된 티켓은 `/gwh:mail-to-ticket` 문제가 아니라 평가자의 실수
