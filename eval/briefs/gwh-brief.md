# Eval Brief: `/gwh:brief`

## 스킬 목적 (주장)
오늘의 **메일 요약 + 캘린더 일정 + (optional) Jira 스프린트 상태**를 통합 브리핑으로 생성한다. triage 캐시가 있으면 재사용한다.

## 해결하겠다는 문제
- 아침에 "오늘 뭘 해야 하지"를 파악하려면 Gmail, Calendar, Jira를 각각 열어야 함 → 3개 도구 컨텍스트 통합
- 시간 대비 큰 그림 조망 (미팅 언제, 블로커 뭐, 액션 뭐)

## 평가 시나리오

### S1. 통합 브리핑 (All Tools Connected)
1. Jira MCP가 연결된 상태에서 `/gwh:brief` 실행
2. 3섹션 확인:
   - 📬 메일 (triage 요약)
   - 📅 오늘 일정
   - 🏃 스프린트 상태
3. 각 섹션이 실제 상황과 맞는가?

### S2. 캐시 재사용
1. `/gwh:triage` 먼저 실행 → `~/.gwh/triage-{today}.md` 생성
2. 이어서 `/gwh:brief` 실행
3. brief가 triage 캐시를 재사용했는지 (Gmail API 추가 호출 없이 빠르게 완료)
4. 실측: cache hit(4ms 수준) vs cache miss(1-2초)

### S3. Jira 미연결 폴백
1. Jira MCP 연결을 끊거나 미사용 상태에서 실행
2. 메일+캘린더 2섹션만 나오는지 (graceful degradation)
3. 에러 없이 코어 기능이 동작하는지

### S4. 일정 충돌 감지
1. 캘린더에 겹치는 미팅이 있는 날 실행
2. 브리핑이 충돌을 감지해 경고(⚠️)하는가?

### S5. 추천 플랜 품질
1. 출력의 "💡 오늘 추천" 섹션 확인
2. 추천 타임블록이 실제 빈 시간대와 일치하는가?
3. 메일 액션과 Jira 티켓의 연결 (시그널 매칭)이 이루어지는가?

## 측정해야 할 지표

| 지표 | 측정 방법 |
|------|----------|
| 섹션 누락 | 3섹션 중 몇 개가 실제로 출력되었나 |
| 일정 정확도 | 출력된 일정 vs Google Calendar 실제 일정 |
| 빈 시간 정확도 | "가용 6.5h" 같은 수치가 실제 빈 시간과 맞는가 |
| 크로스툴 시그널 | 메일 🟡 건수와 Jira In Review 건수가 논리적으로 연결되는가 |

## 루브릭 적용 가이드

- **D1 문제 해결력**: 아침에 이 한 번으로 오늘의 우선순위가 잡히는가?
- **D2 신호/노이즈**: 템플릿 빈칸 vs 진짜 인사이트 비율
- **D3 컨텍스트 절약**: Gmail/Calendar/Jira 각각 여는 5-10분 vs 스킬 1회
- **D4 재사용성**: `~/.gwh/brief-{today}.md`가 standup 보고 자료로 쓸 수 있는가?
- **D5 채택 확률**: 매일 아침 9시에 이걸 실행하는 루틴을 만들 것인가?

## 관찰 포인트

- Jira 연결 시 vs 미연결 시 추천 품질 차이
- "14시에 미팅 2개 겹침" 같은 실제 도움되는 경고가 자동으로 나오는가
- 외부 공유 가능성: 이 브리핑을 복사해서 팀 스탠드업에 쓸 수 있는가
