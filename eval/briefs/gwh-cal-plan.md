# Eval Brief: `/gwh:cal-plan`

## 스킬 목적 (주장)
Jira 스프린트 티켓(또는 수동 입력)을 시간 단위로 분해하여 **Google Calendar 빈 시간에 타임블록으로 배치**한다. 반드시 사용자 승인 후 이벤트 생성.

## 해결하겠다는 문제
- "이번 주 티켓 언제 할까" 고민 → Jira 보고 / 캘린더 보고 / 산수 → 수동 배치 (30분+)
- SP를 시간으로 변환해서 현실성 검증

## 평가 시나리오

### S1. Jira 연결 + 빈 시간 충분 (Happy Path)
1. 스프린트에 티켓 5-8개 할당된 상태
2. `/gwh:cal-plan` 실행
3. 제안 확인:
   - SP → 시간 변환이 합리적인가 (SP 1=1.5h, SP 3=5h 등)
   - 빈 시간대에 정확히 배치되었는가
   - 우선순위(High → Medium) 순서가 지켜졌는가

### S2. 빈 시간 부족 경고
1. 미팅이 많아 가용시간 < 전체 SP일 때
2. 경고(⚠️)가 명확히 표시되는가
3. 부분 배치 + "미배치 태스크 리스트"가 나오는가

### S3. 수동 입력 폴백
1. Jira MCP 미연결 상태에서 실행
2. AskUserQuestion으로 태스크 목록 입력 받는가
3. 수동 입력된 태스크로도 배치가 되는가

### S4. 승인 게이트 검증 (⚠️ 중요)
1. 제안 단계까지 보고 **거절** 선택
2. 캘린더에 실제로 이벤트가 생성되지 않는지 확인 (Google Calendar에서 직접 확인)
3. 산출물 파일(`~/.gwh/cal-plan-{today}.md`)만 저장되는지

### S5. 실제 승인 → 이벤트 생성
1. 작은 범위(1-2개 블록)만 승인
2. `gws calendar +insert`가 실행되어 이벤트가 생성되는가
3. 생성된 이벤트의 제목 포맷이 `[DPPM-XXX] 요약` 형태인가

### S6. 중복 감지
1. 이미 타임블록이 있는 시간대에 또 배치하려 할 때
2. 중복 감지 후 스킵 또는 재배치 선택지 제공하는가

## 측정해야 할 지표

| 지표 | 측정 방법 |
|------|----------|
| SP 추정 정확도 | 배치된 시간 vs 실제 소요 시간 (작업 후) |
| 배치 커버리지 | 전체 SP 중 배치된 비율 |
| 충돌 발생 | 기존 일정과 겹치는 블록 수 |
| 승인 전 생성 사고 | S4에서 거절했는데 이벤트가 만들어지면 **치명적 버그** |

## 루브릭 적용 가이드

- **D1 문제 해결력**: 실제로 주간 계획 수립 시간을 단축하는가?
- **D2 신호/노이즈**: 타임블록 제안이 바로 승인 가능한 수준인가
- **D3 컨텍스트 절약**: 수동 배치 30분+ vs 스킬 사용 시
- **D4 재사용성**: 승인된 플랜이 회고/리뷰 시 참고되는가
- **D5 채택 확률**: 매주 월요일 아침에 이걸 실행할 것인가?

## 관찰 포인트

- SP → 시간 변환 테이블(SP1=1.5h)이 본인 팀 속도와 맞는가
- 캘린더 이벤트 제목에 민감 정보(내부 프로젝트명 등)가 노출되는지
- 승인 게이트가 충분히 명확한가 ("16개 이벤트 생성됩니다" 같은 경고)
- SP 8+ 태스크의 분할 제안이 합리적인가

## ⚠️ 쓰기 스킬 주의사항

이 스킬은 Google Calendar에 **실제 이벤트를 생성**한다. 평가 중:
- 테스트용 캘린더 별도 사용 또는
- 승인 단계에서 거절 후 산출물만 확인
- 실제 이벤트 생성 후에는 수동 삭제 필요 (비대칭 복구 비용)
